Sometimes it is useful to add "tracing logs" to your program so you can see the values of variables and computations as the program is running. For example, if we were writing purely in JavaScript, we might write0 

 
  1. function fib(n) {
  2.   console.log(`Starting to compute Fibonacci`);
  3.   let i = 1;
  4.   let [a, b] = [0, 1];
  5.   while ( i++ < n ) {
  6.     console.log(i, a, b);
  7.     [a, b] = [b, a + b];
  8.   }
  9.   return a;
  10. }
  11. fib(9);

And we'd see the output1

 
  1. Starting to compute Fibonacci
  2. 2 0 1
  3. 3 1 1
  4. 4 1 2
  5. 5 2 3
  6. 6 3 5
  7. 7 5 8
  8. 8 8 13
  9. 9 13 21

How can we do something like this in Reach?2

The key is to use participant interact interfaces to share arbitrary information with the frontend, which has the ability to log to a console or any other tracing service. For example:3 

 
  1. export const main = Reach.App(() => {
  2.   const A = Participant('Alice', {
  3.     logBool: Fun([UBool], Null),
  4.     logNumber: Fun([UInt], Null),
  5.   });
  6.   deploy();
  7.   A.only(() => {
  8.     interact.logBool(true);
  9.     interact.logNumber(1); });
  10.   exit();
  11. });

However, as this example shows, it can be inconvenient to use this pattern, because Fun types constrain their domains to particular input data types, but we may need to log different kinds of data at different points in the program. Similarly, it is inconvenient to use an entire only block for a simple log.4

Reach provides two conveniences for this situation that taste great together: unconstrained domain function types and interact shorthand. The first allows a function in a participant interact interface to have a completely unconstrained domain. The second allows a call to a frontend from anywhere without an only, provided the function returns no value. If we re-write the above example using both of these patterns, it looks like:5 

 
  1. export const main = Reach.App(() => {
  2.   const A = Participant('Alice', {
  3.     log: Fun(true, Null),
  4.   });
  5.   deploy();
  6.   A.interact.log(true);
  7.   A.interact.log(1);
  8.   // We can easily add more complex log entries as well.
  9.   A.interact.log([1, true]);
  10.   A.interact.log({x: 1, y: true});
  11.   const x = 1;
  12.   const y = true;
  13.   A.interact.log({x, y});
  14.   A.interact.log(Maybe(UInt).Some(5));
  15.   exit();
  16. });

Then, a JavaScript frontend can simply use console.log as the value of the log function.6

Reach provides hasConsoleLogger and hasConsoleLogger (Frontend) in the standard library for default implementations of logging to stdout. It can be used in Reach with:7 

 
  1. const A = Participant('Alice', { ...hasConsoleLogger })

and in the JavaScript frontend with:8

 
  1. backend.Alice(
  2.    ctcAlice,
  3.    { ...stdlib.hasConsoleLogger },
  4.  ),

The Reach development repository contains an example of this pattern: log/index.rsh and log/index.mjs.9
